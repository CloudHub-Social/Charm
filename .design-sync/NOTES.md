## Repo shape (why the config looks unusual)
- Charm is a **Tauri app**, not a published component library: no dist, no barrel,
  no `exports`, no `.d.ts` tree. `@/*` → `./src/*` (tsconfig paths + vite alias).
- Storybook is repo-root `.storybook/` (`@storybook/react-vite`); stories are
  `src/components/ui/*.stories.tsx`.

## How discovery is made to work (the app-not-library fix)
- `cfg.entry` = `.design-sync/ds-entry.ts` — a synthesized barrel re-exporting the 9
  storied `ui` modules; esbuild bundles it into `window.Charm` (`cfg.tsconfig`
  resolves `@/`).
- Component discovery needs a real `.d.ts` tree (`exportedNames`/`projectFor` only
  read `.d.ts` and have no path-alias map). `cfg.buildCmd` = `node .design-sync/gen-types.mjs`
  emits declarations (`.design-sync/tsconfig.ds-types.json` → `.design-sync/types/`)
  and writes the barrel `.design-sync/types/ds-entry.d.ts`; **`package.json` `"types"`
  points at that barrel** so discovery finds it. `.design-sync/types/` is gitignored
  (regenerated by buildCmd).

## Overrides
- `overrides`: Dialog/DropdownMenu/Popover/Tooltip → `cardMode: single` (portal/overlay);
  Tabs → `cardMode: column` (Vertical story wider than a grid cell).
- **libOverride `css-fallback.mjs`** (`.design-sync/overrides/`): Charm is dark-first,
  but the generated preview-card chrome (`emit.mjs`) hard-codes a white body, which hid
  light-on-transparent content (input labels, ghost/link buttons, disabled/placeholder
  text). The fork appends a **card-only** dark canvas
  (`.ds-grid,.ds-cell,.ds-single{background-color:var(--color-bg-base)}`) to the scraped
  CSS — those classes exist only in picker cards, never in rendered designs, so designs
  are untouched.
- No `cfg.provider`: dark is the default theme (tokens on `:root`/`body`); `body` already
  carries the canvas bg + `--font-sans`. The `.storybook/preview` theme decorator failed
  to bundle (`.woff2` loader), but it wasn't needed — previews match the storybook
  reference without it (verified via compare).

## CSS / fonts
- No dist CSS → `[CSS_FROM_STORYBOOK]` scrapes the compiled Tailwind v4 + tokens from
  `.design-sync/sb-reference`. Fonts (Manrope, JetBrains Mono via @fontsource) extracted
  to `fonts/`. `[CSS_ASSETS]` warns some `url()` image refs won't resolve post-upload —
  none are images here (fonts copied separately), so it's benign.

## Re-sync risks (watch these next time)
- **Rebuild `.design-sync/sb-reference` whenever a `ui` component or a story changes** —
  it's the compare oracle and does NOT auto-rebuild. Stale reference ⇒ grades compare
  against the OLD design (`[REFERENCE_STALE?]`).
- **`gen-types.mjs` module list must track `ds-entry.ts`** — if a UI component is added
  to the bundle entry, add it to `gen-types.mjs`'s `modules` array too, or it won't be
  discovered.
- **Design palette is limited to compiled classes.** Designs render with the shipped
  `_ds_bundle.css` (Tailwind v4 JIT only emitted *used* classes). `conventions.md` was
  pruned to classes that actually exist in the build (e.g. `bg-input`,
  `text-accent-foreground`, `border-destructive`, `ring-ring` were dropped — present only
  in variant/modifier form). Re-validate the header's class table against the fresh build
  each sync.
- **Story caps:** Button has 8 stories, Avatar 7; the default compare cap captures the
  first 6. The 2026-07-11 canary re-sync graded those 6 each (all `match`); the tail
  stories are verified-by-upload (carried via the anchor). Pass `--max-stories 8`/`7` to
  individually re-grade Button/Avatar's tail. Other components ≤6 (fully graded).
- **`package.json` `"types"` field** exists solely for design-sync discovery; don't
  remove it. It points at a gitignored generated file (`buildCmd` regenerates it).
- **`buildCmd` (`gen-types.mjs`) leaks `.d.ts` into `src-tauri/src/bindings/`.** The
  ui components import IPC bindings via `@bindings/*` → `src-tauri/src/bindings/*.ts`,
  which sit OUTSIDE the ds-types tsconfig `rootDir` (`../src`), so tsc emits their
  declarations in place instead of under `.design-sync/types/`. These untracked
  `src-tauri/src/bindings/*.d.ts` are expected build artifacts — do NOT commit them.
  They live in whichever checkout ran `buildCmd` (the isolated worktree, when you
  synced from one), so clean them **from inside that worktree** —
  `git -C <worktree> clean -f 'src-tauri/src/bindings/*.d.ts'` (or run the clean while
  cd'd into it) — *before* `git worktree remove <worktree>` from the main checkout.
  Cleaning from the main checkout won't touch the worktree's copies, so the plain
  remove still refuses while those untracked files exist.
- **conventions.md color-token naming is loose (not blocking).** The header says the
  utilities "map to a `--color-*` token", but the semantic color vars are actually bare
  shadcn names (`--background`, `--destructive`, …); only `--color-bg-base`/`--color-accent`
  and `--font-*` carry the `--color-*`/`--font-*` prefix. The utility-class table the
  design agent acts on is correct and fully verifies against the build — tighten the
  `--color-*` phrasing only if conventions.md is next edited.

## a11y (storybook gate — informs story authoring)
- `test-storybook:ci` runs axe with `test: error`; only `color-contrast` is globally
  disabled (Spec 09). Overlay stories render `defaultOpen`.
- DropdownMenu stories use `modal={false}` (else the open menu aria-hides the focusable
  trigger → `aria-hidden-focus`). Popover content needs an `aria-label` (Radix gives it
  `role="dialog"` → `aria-dialog-name`).
